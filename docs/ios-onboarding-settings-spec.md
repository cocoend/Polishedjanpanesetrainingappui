# iOS 欢迎引导页、登录页与设置页设计/功能文档

> 本文档中的页面内容已整合进 `docs/ios-page-design-spec.md`。
> 如需查看当前完整 iOS 页面设计，请优先参考总文档。

## 1. 背景与目标

当前 `PolishedJapaneseTraining` iOS App 已具备练习主流程：

- 首页 `HomeScreen`
- 主题选择/详情
- 模型选择/介绍
- 录音
- 反馈
- Learned Box

本次需要补齐的能力有三类：

1. 首次启动的新手欢迎与使用引导
2. 第 0 页情绪欢迎页
3. 第 4 页登录页
3. 设置页与“游客受限能力拦截”

本次方案目标：

- 帮助新用户快速理解 App 的价值和使用方式
- 提供“登录”与“暂不登录”的双路径
- 允许用户先以游客模式体验核心练习流程
- 当游客触发保存、同步等用户型功能时，引导其先登录
- 保持与当前首页一致的温和、鼓励型、卡片式视觉语言

## 2. 产品原则

### 2.1 使用模式

App 支持两种用户身份：

- 登录用户
- 游客用户

两者规则如下：

- 登录用户：可使用完整用户功能
- 游客用户：可使用浏览、选题、录音、获取反馈等核心体验
- 游客用户：一旦触发需要“账户承载”的功能，必须先去登录
- 特别规定：游客在反馈页点击“保存到 Learned Box”，或执行“将反馈保存到 box”这类动作时，必须立即跳转登录拦截

### 2.2 受限功能原则

以下功能默认视为“需要登录”：

- 保存到 Learned Box
- 查看个人 Learned Box 历史记录
- 删除个人云端数据
- 后续若增加跨设备同步、连续学习记录、提醒同步，也默认需要登录

### 2.3 引导原则

- 第 0 页用于建立第一眼好感
- 第 1 到 3 页用于建立理解和信任
- 第 4 页用于完成身份选择
- “暂不登录”必须可见、可点、弱化但不隐藏
- 游客模式不是阉割版首页，而是“可先试用、关键能力再登录”

## 5. 首次启动流程

### 5.1 Onboarding 页数调整

引导页调整为 5 页：

0. 情绪欢迎页
1. 产品欢迎页
2. 如何使用
3. 权限说明
4. 登录页

### 5.2 进入规则

- 首次启动：必须进入 5 页引导
- 非首次启动：直接进入首页
- 用户在设置页中可再次查看前 3 页介绍
- 第 4 页登录页不建议作为“再次查看引导”的一部分重复出现

原因：

- 第 0 页是情绪欢迎页，不承担功能教育
- 第 4 页不是纯介绍，而是首次身份选择页
- 后续再次登录应从设置页、受限能力拦截页进入

## 6. 本地状态与身份状态

建议新增以下状态：

- `hasCompletedOnboarding: Bool`
- `isLoggedIn: Bool`
- `isGuestMode: Bool`
- `hasSeenMicrophonePermissionIntro: Bool`
- `preferredDailyReminderEnabled: Bool`
- `preferredReminderTime: Date`
- `autoSaveToLearnedBox: Bool`
- `pendingProtectedAction: ProtectedAction?`

其中：

- `isLoggedIn == true` 时，`isGuestMode == false`
- `isGuestMode == true` 时，代表用户明确选择过“暂不登录”
- `pendingProtectedAction` 用于记录游客点击了哪个受限功能，登录成功后可继续执行

建议新增枚举：

```swift
enum ProtectedAction: Hashable {
    case saveToLearnedBox
    case openLearnedBox
    case deleteMyData
}
```

## 7. 存储建议

优先使用 `@AppStorage` 或 `UserDefaults` 保存：

- `hasCompletedOnboarding`
- `isGuestMode`
- 本地偏好项

说明：

- 当前项目已经使用 `UserDefaults` 保存匿名用户 ID
- 在登录系统尚未完全接入前，游客态可继续沿用本地匿名标识做临时会话
- 一旦后续接入真实登录，需要明确“匿名态数据是否迁移到登录账号”

当前文档建议：

- 第一阶段先不做游客数据迁移
- 游客只允许体验，不承诺云端保存能力

## 8. RootView 入口控制建议

建议在 `RootView` 或 `AppState` 增加启动态判断：

- `hasCompletedOnboarding == false` 时展示 onboarding flow
- `hasCompletedOnboarding == true` 时展示现有主业务 `NavigationStack`

推荐额外增加一个入口状态层，例如：

- `AppEntryState`

可包含：

- onboarding 是否完成
- 当前是否登录
- 当前是否游客模式
- 是否需要展示登录拦截弹窗

## 9. 五页引导详细设计

## 9.0 第 0 页：情绪欢迎页

### 目标

只做一件事：
让用户一打开 App 就感觉亲切、可爱、轻松，愿意继续看下去。

这页不承担复杂功能说明，不讲权限，不讲登录，不讲完整流程。

### 页面内容

- 大面积留白或柔和渐变背景
- 可爱、轻松的插画或拟物元素
- 简单打招呼文案
- 一句很短的欢迎语
- 底部一个明确按钮：`开始看看`

### 文案建议

可以是偏日语/中日混合的轻问候风格，例如：

- `こんにちは`
- `欢迎来到 Polished`
- `今天也来轻松练一小段日语表达吧`

### 视觉方向

- 氛围优先，不要信息太多
- 颜色建议更柔和一些，可用浅橙、浅绿、奶白色
- 允许这页比后面几页更“可爱”、更感性
- 可以加入角色、对话气泡、漂浮元素，但不要过于幼态

### 交互规则

- 点击 `开始看看` -> 进入第 1 页

### 设计建议

- 这是一个“开门页”
- 更像品牌欢迎，而不是产品教程
- 如果后续要做动画，这页最适合做轻微呼吸感或漂浮感动效

## 9.1 第 1 页：欢迎页

### 目标

让用户立刻明白：
这是一个帮助用户练习“日语说明能力”的 App。

### 页面内容

- 顶部欢迎视觉区
- 主标题：`欢迎开始你的日语说明训练`
- 副标题：`把“知道怎么说”练成“自然说出来”`
- 三个价值点：
  - AI 帮你分析表达
  - 从日常主题开始练习
  - 积累自己的表达素材
- 底部按钮：
  - 主按钮：`继续`
  - 次按钮：`跳过`

### 交互规则

- `继续` -> 第 2 页
- `跳过` -> 第 3 页

## 9.2 第 2 页：使用介绍页

### 目标

用最短路径让用户理解核心使用流程。

### 页面内容

- 标题：`3 步开始一次练习`
- 步骤 1：选择主题
- 步骤 2：录音表达
- 步骤 3：获得 AI 反馈
- 辅助提示：
  `登录后还可以把满意的反馈保存到 Learned Box`
- 按钮：
  - 主按钮：`我明白了`
  - 次按钮：`返回`

### 交互规则

- `我明白了` -> 第 3 页
- `返回` -> 第 1 页

## 9.3 第 3 页：权限说明页

### 目标

在麦克风权限弹窗前解释录音用途，降低拒绝率。

### 页面内容

- 标题：`开始前，需要录音权限`
- 说明卡片：
  - 仅在你主动开始练习时录音
  - 录音将用于文字转写与 AI 反馈
  - 你可以在设置中查看隐私说明
- 状态文案：
  - 未授权：`授权后即可开始练习`
  - 已拒绝：`你可以稍后去系统设置重新开启权限`
- 按钮：
  - 主按钮：`允许并继续`
  - 次按钮：`稍后再说`

### 交互规则

- `允许并继续`
  - 触发权限请求
  - 不管授权成功与否，都进入第 4 页登录页
- `稍后再说`
  - 直接进入第 4 页登录页

说明：

- 权限说明页不再作为 onboarding 终点
- 第 4 页登录页才是 onboarding 的最终收口页

## 9.4 第 4 页：登录页

### 目标

让用户完成首次身份选择：

- 登录进入完整模式
- 暂不登录，以游客模式先体验

### 页面内容

- 顶部欢迎文案
  - 主标题：`登录后，保存你的练习进度`
  - 副标题：`登录可保存到 Learned Box，并在后续获得完整用户功能`
- 中部权益卡片
  - 保存练习记录
  - 查看个人 Learned Box
  - 后续支持跨设备同步
- 主按钮
  - `登录 / 注册`
- 次级文本入口
  - 一行灰色 URL 风格文案：`暂时不登录，先以游客模式体验`

### 视觉要求

- 主按钮高对比、醒目
- “游客模式”必须是灰色、细字、URL 风格
- 不可比主按钮更抢眼
- 但点击热区要足够大，避免误触困难

### 交互规则

- 点击 `登录 / 注册`
  - 进入正式登录流程
  - 登录成功后：
    - `isLoggedIn = true`
    - `isGuestMode = false`
    - `hasCompletedOnboarding = true`
    - 进入首页
- 点击 `暂时不登录，先以游客模式体验`
  - `isLoggedIn = false`
  - `isGuestMode = true`
  - `hasCompletedOnboarding = true`
  - 进入首页

## 10. 游客模式规则

### 10.1 游客可用功能

- 浏览首页
- 浏览主题
- 选择说明模型
- 开始录音
- 获得 AI 反馈

### 10.2 游客不可用功能

- 保存到 Learned Box
- 将反馈保存到 box
- 打开 Learned Box 历史记录
- 删除个人云端数据
- 未来所有“需要账户归档/同步”的功能

### 10.3 触发策略

游客点击受限功能时，不应静默失败，也不应只 toast 一句提示。

统一策略：

- 弹出登录拦截页或底部 sheet
- 不允许先本地保存、后补登录，也不允许静默失败
- 清晰说明为什么需要登录
- 提供：
  - 主按钮：`登录后继续`
  - 次按钮：`取消`
  - 辅助链接：`继续游客体验`

## 11. 登录拦截设计

### 11.1 触发场景

至少包括以下场景：

1. 游客点击反馈页“保存到 Learned Box”
2. 游客点击“保存反馈到 box”或任何同义保存动作
3. 游客点击首页右上角进入 Learned Box
4. 游客在设置页点击“删除我的数据”

### 11.2 拦截文案建议

标题：

- `登录后即可使用此功能`

正文根据场景动态变化：

- 保存场景：
  `保存练习反馈到 Learned Box 需要登录。游客点击保存时，必须先进入登录流程，这样记录才能被安全保存和找回。`
- 打开 Learned Box 场景：
  `Learned Box 用于查看你保存的练习记录，需要登录后使用。`
- 删除数据场景：
  `数据删除涉及你的个人练习记录，需要先确认登录身份。`

### 11.3 登录成功后的续接

若拦截前记录了 `pendingProtectedAction`，登录成功后应自动继续用户原操作：

- `saveToLearnedBox` -> 自动继续保存
- `openLearnedBox` -> 自动跳转 Learned Box
- `deleteMyData` -> 回到确认删除流程

这样可以减少用户“刚决定登录，结果还得再点一遍”的中断感。

## 12. 设置页调整建议

由于引入登录与游客模式，建议将“账号管理”从设置页中单独拆成用户页。

设置页结构建议调整为 5 个 section：

1. 账号入口
2. 练习偏好
3. 提醒与通知
4. 帮助与说明
5. 隐私与数据

## 12.1 Section A：账号入口

### 登录用户状态

展示：

- 头像占位
- 昵称/邮箱/手机号占位
- 状态：`已登录`

点击后进入 `用户页 UserProfileScreen`

### 游客用户状态

展示：

- 标题：`当前为游客模式`
- 说明：`登录后可保存练习记录与使用完整功能`
- 主按钮：`立即登录`

游客状态下不进入完整用户页，点击后直接进入登录流程。

## 12.2 用户页设计

用户页建议作为登录后的独立页面，而不是把所有账号操作都塞进设置首页。

### 页面定位

- 承载个人资料
- 承载账号安全
- 承载绑定关系
- 承载登出与账号相关操作

### 页面入口

- `Home -> Settings -> 账号卡片/头像区 -> UserProfileScreen`
- 或 `Settings` 顶部直接展示用户卡片，点击进入

### 页面结构

建议分为 5 个 section：

1. 用户头部信息
2. 个人资料
3. 账号绑定与安全
4. 学习账号权益说明
5. 退出登录

### Section 1：用户头部信息

展示内容：

- 头像
- 昵称
- 用户 ID 或简化账号标识
- 登录状态：`已登录`

可选补充：

- 学习天数
- 已保存到 Learned Box 的数量

页面风格建议：

- 顶部大卡片
- 延续 `HomeScreen` 的柔和渐变视觉
- 使用较强的身份感，让用户知道这里是“我的账户”

### Section 2：个人资料

建议包含以下项目：

- `编辑用户信息`
- `修改昵称`
- `修改头像`

#### 编辑用户信息

点击进入资料编辑页或 sheet。

建议可编辑字段：

- 昵称
- 头像

第一阶段建议仅保留：

- 昵称
- 头像

### Section 3：账号绑定与安全

这是用户页的重点模块，必须包含你提到的邮箱绑定能力。

建议包含：

- `绑定邮箱`
- `修改绑定邮箱`
- `邮箱验证状态`
- `修改密码`（若使用密码登录）

#### 绑定邮箱

展示状态分两种：

- 未绑定：`未绑定邮箱`
- 已绑定：`name@example.com`

操作规则：

- 未绑定时，按钮文案：`绑定邮箱`
- 已绑定时，按钮文案：`更换绑定邮箱`

交互建议：

- 点击后进入邮箱输入页
- 输入邮箱后发送验证码或验证邮件
- 完成验证后更新状态为已绑定

#### 邮箱验证状态

状态建议：

- `未验证`
- `已验证`

若未验证：

- 展示辅助按钮：`重新发送验证邮件`

#### 账号安全

若当前产品后续要接密码体系，建议预留：

- `修改密码`

若第一阶段暂未接入，可先在文档中标注“预留”。

### Section 4：学习账号权益说明

这一块不是必须，但很适合放在用户页中部，用来强化登录价值。

建议内容：

- 保存反馈到 Learned Box
- 查看个人练习历史
- 后续支持多设备同步

作用：

- 让用户知道账号不是“为了注册而注册”
- 让游客转登录后感知到权益闭环

### Section 5：退出登录

这一项必须有，而且要单独强调为危险操作级别较低但结果明确的动作。

展示方式建议：

- 单独一组 section
- 底部按钮：`退出登录`
- 红色文字或警示样式，但不要和“删除账号/删除数据”混淆

交互规则：

- 点击 `退出登录`
  - 弹出二次确认框
  - 文案：`退出后，你将回到游客模式，但不会删除已保存的账号数据`
- 用户确认后：
  - `isLoggedIn = false`
  - `isGuestMode = true`
  - 返回首页或设置页

### 用户页不建议混入的内容

以下内容不建议放在用户页，仍建议留在设置页：

- 麦克风权限
- 通知提醒
- 隐私政策
- 删除我的数据
- App 版本信息

### 学习目标 / 日语水平的放置建议

`学习目标` 与 `日语水平` 不再放在用户页中。

改为放在首次登录成功后的引导问卷中采集，更适合当前产品流程。

原因：

- 这两项更偏“首次建档信息”，而不是日常账号管理信息
- 放在首次登录引导中更容易形成完整画像
- 可以直接服务后续首页推荐、主题推荐和学习路径个性化

建议问卷时机：

- 用户第一次成功登录后
- 首次进入首页前，或首次进入首页后以轻量弹层形式出现

建议问卷字段：

- 当前日语水平：初级 / 中级 / 高级
- 学习目标：日常交流 / 面试表达 / 留学沟通 / 工作汇报 / 其他

问卷原则：

- 可跳过
- 问题数量控制在 2 到 3 个
- 提交后可用于首页推荐，但后续不强制在用户页编辑

## 12.3 Section B：练习偏好

保留以下方向：

- 自动保存优秀反馈到 Learned Box
- 默认提醒偏好

但规则要改为：

- 游客看到该开关时，可展示但禁用，或点击后要求登录

推荐：

- 若功能实际依赖账号，则显示为禁用态并附文案 `登录后可开启`

## 12.4 Section C：提醒与通知

可保留：

- 每日练习提醒 `Toggle`
- 提醒时间

若未来提醒需要跨设备同步，则登录后体验更完整；
但第一阶段可以先允许游客使用本地提醒。

## 12.5 Section D：帮助与说明

建议包含：

- 查看新手介绍
- 如何使用 Learned Box
- 反馈是如何生成的

注意：

- 再次查看引导时，默认仅展示前 3 页说明内容
- 不重复展示第 4 页首次登录选择页

## 12.6 Section E：隐私与数据

建议包含：

- 麦克风权限状态
- 隐私说明
- 删除我的数据
- 当前版本

其中“删除我的数据”规则：

- 登录用户：正常进入确认删除流程
- 游客用户：先触发登录拦截

## 13. 首页入口调整建议

首页右上角建议保留两个入口：

- Learned Box
- Settings

其中：

- 游客点击 Learned Box -> 直接弹登录拦截
- 登录用户点击 Learned Box -> 正常进入

Settings 不应受限：

- 游客也应该能进入设置页
- 这样用户能在设置页里看到“立即登录”和帮助说明

## 14. 文案建议

### 14.1 登录页文案语气

建议方向：

- 鼓励但不逼迫
- 强调“登录后的好处”
- 不制造“必须注册才能用”的压迫感

示例：

- `登录后可保存你的练习记录`
- `先用游客模式体验也可以`

避免：

- `必须登录才能继续`
- `不登录将无法使用本应用`

### 14.2 游客拦截文案语气

建议：

- 解释限制原因
- 说明登录收益
- 给返回路径

例如：

- `这个功能需要登录后使用，这样我们才能帮你安全保存练习记录。`

## 16. 验收标准

### Onboarding 验收

- 首次启动时必定出现 5 页
- 第 0 页为纯欢迎页，只承担情绪开场作用
- 第 4 页必须同时提供登录与游客模式
- 游客模式入口显示为一行灰色 URL 风格文字
- 选择任一路径后都能正确进入首页

### 游客拦截验收

- 游客点击“保存到 Learned Box”时，必须跳到登录拦截
- 游客点击“将反馈保存到 box”时，必须跳到登录拦截
- 游客点击 Learned Box 时，必须跳到登录拦截
- 游客点击删除数据时，必须跳到登录拦截
- 登录成功后能续接用户刚才的目标操作

### 设置页验收

- 游客可进入设置页
- 设置页能正确展示“游客模式”状态
- 登录用户可展示“已登录”状态
- 帮助与隐私入口可正常打开

## 17. 分阶段实施建议

### Phase 1

- 第 0 页欢迎页
- 五页 onboarding
- 第 4 页登录/游客分流
- 首页 settings 入口
- 游客点击 Learned Box 拦截
- 游客点击保存到 Learned Box 拦截
- 设置页账号 section

### Phase 2

- 登录成功后续接原操作
- 删除数据的登录态治理
- 自动保存逻辑与账号联动
- 更完整的账号信息页

## 18. 结论

这次改动的重点已经从“补页面”升级为“建立情绪欢迎 + 建立身份体系”：

- 第 0 页负责建立第一眼好感
- 第 1 到 3 页负责建立产品理解
- 第 4 页负责完成身份选择
- 游客模式负责降低首次体验门槛
- 登录拦截负责把关键用户功能收口到账号体系
- 设置页负责承接账号、帮助、偏好、隐私与数据管理

按当前项目结构，这套方案与现有 `HomeScreen`、`AppState`、`FeedbackScreen`、`LearnedBoxScreen` 的衔接点都比较明确，适合直接拆成后续设计稿与 iOS 开发任务。
